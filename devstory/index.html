PROLOGUE

왜 만들었나

경제 뉴스를 꾸준히 보고 싶었습니다. 그런데 매일 여러 사이트를 돌아다니기가 번거로웠습니다. "AI가 알아서 정리해주면 안 될까?" — 이 질문 하나에서 시작됐습니다.

💡

바이브코딩(Vibe Coding)이란?

코드를 직접 짜는 대신, AI에게 원하는 것을 말로 설명하고 AI가 코드를 작성하는 개발 방식입니다. 기획자나 비개발자도 실제 작동하는 서비스를 만들 수 있습니다. 이 앱은 처음부터 지금까지 코드를 직접 작성한 적이 없습니다.

📰

처음 목표

경제/산업/국제 뉴스를 AI가 핵심만 골라 짧게 정리해주는 앱. 매일 아침 5분 안에 뉴스를 파악할 수 있도록.

🛠

개발 도구

Claude (AI 개발 파트너), GitHub (코드 저장소), Vercel (배포), Naver API (뉴스), Anthropic API (AI 요약).

🎯

핵심 원칙

비용 최소화 · 빠른 로딩 · 정확한 정보 — 이 세 가지를 기준으로 모든 기술 결정을 내렸습니다.

v20~v40

문제 발생 — 테스트할 때마다 돈이 나간다

글자 색 하나 바꾸는 테스트에도 Claude API가 호출됐습니다. 하루 만에 예상치 못한 API 비용이 청구됐습니다. 대책이 필요했습니다.

Before
  • UI 변경 → 반드시 실제 API 호출
  • 테스트 1회 = 비용 발생
  • 하루 수십 번 테스트 → 예상 외 청구
→
After (DEV MODE)
  • ?dev=true로 가짜 데이터 사용
  • API 호출 없이 UI만 확인
  • 개발 중 AI 비용 = 0원
API 비용 문제 DEV MODE 도입 ?dev=true 파라미터
v50~v70

Naver API 429 오류 — 요청이 너무 빠르다

뉴스를 한꺼번에 가져오려 하니 Naver에서 "요청이 너무 많다"며 차단(429 오류). 배치 처리(3개씩, 0.25초 간격)로 해결.

429 Too Many Requests 배치 처리 (3개씩 × 250ms)
v70~v100

localStorage 캐시 — 같은 데이터를 두 번 부르지 않는다

사용자가 탭을 닫고 다시 열어도 일정 시간 동안 저장된 데이터를 재사용. 하루 2회 고정 스케줄(07:30, 17:00) 이후에만 새로 가져오도록 설계.

Before
  • 접속할 때마다 API 호출
  • 같은 날 10번 접속 = 10번 과금
→
After
  • localStorage에 결과 저장
  • 스케줄 시간 전이면 재사용
  • 하루 접속 100번 = API 호출 2번
localStorage 캐시 스케줄 기반 무효화 하루 2회 갱신

v105

Haiku 모델 실패 — 뉴스 방향을 반대로 요약

비용 절감을 위해 가장 저렴한 Claude Haiku 모델로 브리핑 생성을 시도했습니다. 결과: 뉴스 내용을 사실과 정반대로 요약하는 사고 발생. 경제 뉴스에서 방향이 틀리면 투자 판단 오류로 이어집니다.

Haiku (저렴)
  • 비용: 저렴
  • 속도: 빠름
  • 정확도: 낮음 — 사실 오류 발생
  • 경제 뉴스에 부적합
→
Sonnet (균형)
  • 비용: 적당
  • 정확도: 높음
  • 경제 요약에 충분한 성능
  • 주력 브리핑 모델로 고정
Haiku 정확도 실패 Sonnet으로 고정 Haiku는 댓글 생성 등 보조로만
v110~v120

프롬프트 엔지니어링 — AI에게 정확히 지시하는 법

AI의 품질은 "어떻게 물어보는가"에 달려있다는 걸 실감. 환각(없는 내용 창작) 방지를 위해 "제공된 뉴스 외 정보 금지" 명시. 마크다운 금지, 파싱 태그 강제([SUMMARY]...[/SUMMARY]), 줄1:/줄2:/줄3: 레이블 규칙 확립.

안티-환각 프롬프트 [SUMMARY] 파싱 태그 마크다운 출력 금지

→
After (Firebase)
  • 모든 사용자 즉시 로딩
  • 서버에 저장 → 기기 무관
  • 미리 생성된 데이터 전달

설계 첫 설계가 미래 비용을 결정한다

"일단 작동하게"로 시작해도 되지만, 캐시 전략만큼은 초기에 잘 잡아야 합니다. v70에서 localStorage 캐시를 도입했을 때, v131에서 Firebase로 전환할 때 — 캐시 구조 변경은 항상 가장 큰 작업이었습니다. 처음부터 "누가, 언제, 어디서 생성하고 누가 읽어가는가"를 명확히 했더라면 리팩터링이 훨씬 적었을 것입니다.

비용 싼 모델은 진짜로 위험할 수 있다

Haiku로 브리핑을 생성했을 때 뉴스 방향이 반대로 나왔습니다. 투자 뉴스에서 방향이 틀리면 피해는 사용자에게 갑니다. 비용 절감과 정확도 사이에는 트레이드오프가 있고, 콘텐츠 서비스에서는 정확도가 우선입니다. Haiku는 댓글처럼 "틀려도 괜찮은 영역"에서만 씁니다.

AI 활용 AI에게 지시하는 능력이 결과물의 품질을 결정한다

바이브코딩은 코드를 몰라도 됩니다. 하지만 "무엇을 원하는지 정확히 말하는 능력"은 필수입니다. 프롬프트가 불명확하면 AI는 그럴듯하지만 잘못된 결과를 자신 있게 냅니다. 150번 버전을 올리면서 가장 크게 성장한 것은 코딩이 아니라 정밀한 요구사항 정의였습니다.

협업 AI는 제안하고, 사람이 결정한다

AI는 최선이라 생각하는 방향으로 코드를 쏟아냅니다. 특히 상위 모델일수록 확신이 강합니다. 하지만 방향을 정하는 것은 사람의 역할입니다. "모든 변경은 먼저 보고하고, 합의 후 진행" — 이 원칙 하나가 예상치 못한 변경과 비용 충격을 막아줍니다.

유지보수 기능 추가보다 정리가 더 중요할 때가 있다

v148~v151은 새 기능이 거의 없습니다. 죽은 코드 제거, console.log 정리, 중복 계산 통합, 크론 단순화. 겉으로 보이지 않는 변화지만, 이런 정리가 쌓여야 다음 단계로 나아갈 수 있습니다. "일단 작동하면 됐다"는 생각이 쌓이면 나중에 건드리기 두려운 코드가 됩니다.
